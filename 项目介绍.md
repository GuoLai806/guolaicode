# 项目介绍

## 项目概述

GuoLaiCode 是一个功能丰富的终端 AI 编码助手，基于 Python 3.11+ 开发，使用 Textual 作为 TUI 框架。

## 核心功能

| 功能 | 说明 |
|------|------|
| 多模型支持 | Anthropic, OpenAI, OpenAI兼容协议（如豆包） |
| 多Agent协作 | 支持子Agent, 团队协作 |
| 技能系统 | 可扩展的自定义技能 |
| 权限控制 | 5层权限防御链 |
| 上下文管理 | 自动压缩, 长期记忆 |
| MCP集成 | 支持Model Context Protocol |

## 🔒 为什么目前没有沙箱？

我们经常被问到：「GuoLaiCode 为什么没有像 Claude Code 那样的沙箱？」，这里是完整的技术说明：

### 1. 什么是沙箱？

在AI编码助手语境下，「沙箱」通常有两种含义：

| 类型 | 说明 | 实现难度 |
|------|------|---------|
| **软件安全层（GuoLaiCode 已有）** | 正则黑名单, 路径检查, 权限规则 | ✅ 已实现 |
| **真实容器/虚拟机隔离（Claude Code 才有）** | bubblewrap, Docker, 虚拟机 | ⚠️ 复杂 |

### 2. GuoLaiCode 已有完整的软件安全层

GuoLaiCode 虽然没有容器隔离，但已经实现了**5层纵深防御链**：

```python
# GuoLaiCode 的权限防御链
1. Plan Mode 例外（只读模式只允许特定操作）
2. 安全命令白名单（ls/pwd/git status 等自动放行）
3. 危险命令黑名单（rm -rf /, mkfs, 管道脚本等直接拦截）
4. PathSandbox 路径检查（限制只能访问项目目录和系统临时目录）
5. 规则引擎（用户/项目/本地三层规则）
6. 权限模式（default/accept-edits/plan/bypass）
7. HITL 人工确认（最终由用户确认）
```

对于**个人开发/学习用途**，这套安全方案已经足够安全。

### 3. 为什么没有加真实的容器沙箱？

| 原因 | 详细说明 |
|------|---------|
| **平台兼容性问题** | Claude Code 只在 macOS/Linux/WSL2 支持沙箱，Windows 完全没有沙箱支持。GuoLaiCode 是跨平台项目，如果加沙箱，Windows 怎么办？ |
| **技术复杂度高** | 真实沙箱（如 Claude Code 使用的 bubblewrap + seccomp）需要大量底层系统集成，对一个开源学习项目来说，投入产出比不高 |
| **依赖管理复杂** | 需要安装额外系统组件（bubblewrap, socat, seccomp），对于普通用户门槛太高 |
| **项目定位** | GuoLaiCode 主要用于学习和个人开发，不是企业级生产工具，5层防御链已经满足需求 |
| **Claude Code 本身也不是全平台沙箱** | 即使是 Claude Code，在 Windows 上也没有沙箱，可见这不是一个能简单解决的问题 |

### 4. PathSandbox 是真的沙箱吗？

**不是！** 但它很有用：

```python
# PathSandbox 核心实现
class PathSandbox:
    def check(self, path):
        # 真实路径解析（防止 symlink 绕过）
        real_path = resolve(path)
        
        # 检查是否在允许根目录下
        for root in self._allowed_roots:
            try:
                real_path.relative_to(root)
                return True
            except ValueError:
                continue
        return False
```

**⚠️ 关键理解：**
- PathSandbox 是**软件验证层**，不是**隔离层**
- 它会**拦截访问**，但不会**隔离环境**
- 如果命令绕过检查，会直接在真实环境执行！

### 5. 想要沙箱怎么办？

如果你**确实需要容器级别的隔离**，有几个替代方案：

#### 方案 1：自己在 Docker 里运行 GuoLaiCode
```bash
# 把整个项目放进 Docker 容器里运行
# 这样即使出问题也是容器内的问题
```

#### 方案 2：使用工作树模式（Git Worktree）
GuoLaiCode 有 Git Worktree 集成，每个任务在独立分支里进行，一定程度上减少了误操作风险。

#### 方案 3：修改代码，添加 Docker 包装层（贡献欢迎！）
如果你愿意贡献代码，可以参考 Claude Code 的架构，给 GuoLaiCode 添加沙箱支持。

## 总结

- GuoLaiCode **没有容器/虚拟机沙箱隔离**
- 但有**完整的5层权限防御链**
- 对于学习和个人开发，已经足够安全
- 如果需要企业级安全，建议自己在 Docker 中运行，或贡献代码给 GuoLaiCode！
