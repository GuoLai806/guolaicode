你还记得第 1 章结束时的感觉吗 ？

那时候 GuoLaiCode 还只是一个想法。你知道了 Agent 的四要素是 LLM + 工具 + 循环 + 反馈，知道了 Coding Agent 为什么是学 Agent 的最佳沙盘，选好了编程语言和 LLM API，装好了开发环境。但代码一行都没写，终端里什么都跑不起来。

而现在，十五章之后，你手里的 GuoLaiCode 能自己读代码、写文件、跑命令、搜索代码库。它有权限系统防止误操作，有 MCP 协议接入外部工具生态，有上下文压缩控制成本，有记忆系统跨会话延续经验。它能接受用户自定义命令，能加载可复用的技能包，能在生命周期的关键节点触发自动化钩子。它甚至能拉起一支团队，把任务拆给多个 Agent 并行协作。

这一章，我们不写新代码。我们站远一点，回头看看这个 Agent 是怎么从一个想法变成一个完整系统的。

***

## 第一声啼哭：让 AI 开口说话

每个程序的起点都很不起眼。GuoLaiCode 也一样。

第 1 章没有写任何代码。它做的事情更基础：搞清楚我们到底要造什么。Agent 的四要素是 LLM + 工具 + 循环 + 反馈，Coding Agent 之所以适合学 Agent 架构，是因为代码输出可以自动验证，天然形成反馈循环。这些概念当时可能觉得抽象，但后面每一章你都在反复印证它们。

第 2 章才是 GuoLaiCode 真正写下第一行代码的时刻。接上 LLM API，发一条消息，收到一条回复。就这么简单的一个来回，却是整个系统的起点。

你可能已经忘了当时的兴奋感了。第一次看到终端里的文字不是你打的，是 AI 一个字一个字蹦出来的，那种感觉很奇妙。流式响应让文字像打字机一样跳出来，状态栏实时显示 token 消耗，多轮对话让它能记住上文。

![]()

但说实话，那时候的 GuoLaiCode 跟一个聊天机器人没什么区别。你问它一个问题，它回你一段话。一问一答，仅此而已。它不能读你的代码，不能改你的文件，不能执行任何命令。

就像一个刚出生的婴儿，会哭了，但还不会走路。

***

## 学会用手：工具系统和 Agent Loop

第 3 章给 GuoLaiCode 装上了六个工具：ReadFile、WriteFile、EditFile、Bash、Glob、Grep。这六个工具覆盖了一个程序员日常工作的基本动作：读文件、写文件、改文件、跑命令、找文件、搜内容。

工具本身的实现并不复杂。每个工具就是一个函数，接收参数，执行操作，返回结果。真正让人兴奋的是 function calling 协议。你不需要教 AI 怎么解析命令行参数，不需要设计一套自己的指令格式。你只要把工具的 JSON Schema 描述告诉 Claude，它就知道什么时候该用什么工具，参数怎么填。

![]()

但装了工具还不够。那时候的 GuoLaiCode 每次只能做一步。模型返回一个工具调用，你执行完返回结果，模型给一个最终回复，就结束了。你得一步步催它。

第 4 章解决了这个问题。Agent Loop，一个 while 循环，让 GuoLaiCode 学会了自己干活。

这个循环的逻辑简单到让人不敢相信：调 LLM，看返回里有没有 tool\_ use，有的话执行工具、把结果喂回去、再调一次 LLM。没有 tool\_ use 了，说明任务完成，退出循环。就这样。

![]()

但就是这么简单的一个 while 循环，让 GuoLaiCode 从一个聊天工具变成了一个真正的 Agent。你跟它说一句「帮我写一个 HTTP 服务器」，它自己扫项目结构，自己读现有代码，自己写新文件，自己编译检查，发现报错了自己改，改完再编译，直到通过为止。

回想第 1 章定义的 Agent 四要素：LLM、工具、循环、反馈。到第 4 章结束，四个格子全填满了。GuoLaiCode 在这个时刻从概念变成了现实。

![]()

***

## 知道自己是谁：System Prompt

到第 4 章结束，GuoLaiCode 能自主干活了。但它还不知道自己是谁。

你让它写代码，它会写。但用什么风格？遇到不确定的事情该谨慎还是大胆？跟用户说话该正式还是随意？它应该主动解释自己在做什么，还是闷头干完才汇报？这些问题，Agent Loop 回答不了，工具系统也回答不了。

第 5 章用 System Prompt 解决了这个问题。System Prompt 是每次对话开始前注入给 LLM 的第一条指令，定义了 Agent 该扮演什么角色、遵守什么规则、用什么方式跟用户交互。它不参与 Agent Loop 的运转，但它塑造了循环中每一次 LLM 决策的倾向。

![]()

你可以把 System Prompt 想象成一个人的职业素养。一个实习生和一个资深工程师，拿到同样的工具，做出来的东西完全不同。不是因为工具不同，而是因为经验、判断力和行为准则不同。System Prompt 就是给 GuoLaiCode 注入这些「软实力」的地方。同一个 Agent Loop，换一套 System Prompt，就能表现得像完全不同的 Agent。

***

## 装上刹车，打开大门

一个能自主干活的 Agent 听起来很酷，但你真的敢让它在你的电脑上为所欲为吗？

它可以执行 Bash 命令。那如果它决定跑一个 `rm -rf /` 呢？它可以写文件。那如果它把你的配置文件覆盖了呢？能力越大，风险越大。一个没有安全约束的 Agent 就像一辆没有刹车的跑车，速度再快也没人敢坐。

![]()

第 6 章给 GuoLaiCode 装上了五层安全防护。

最底层是危险命令检测，用正则表达式拦截已知的破坏性命令。往上是路径沙箱，防止 Agent 通过符号链接逃逸到项目目录之外。

再往上是多层规则引擎，用户、项目、本地三级配置叠加，精细控制每个工具的权限。然后是 四种权限模式 ，从完全信任到完全不信任，用户按需选择。最顶层是 HITL 确认机制，遇到敏感操作时暂停循环，让人类做最终决定。

这五层层层叠加，漏过了一层，还有下一层兜底。就像汽车的安全系统：ABS、安全带、安全气囊、碰撞预警，每个都独立工作，加在一起才构成完整的保护。

![]()

有了刹车，接下来要打开大门。

第 7 章引入了 MCP 协议。MCP 的全称是 Model Context Protocol，它让 GuoLaiCode 不再局限于内置的六个工具。通过 MCP，任何外部服务都可以把自己包装成工具提供给 Agent 使用。数据库查询、API 调用、文档搜索、日历管理，只要有人写了 MCP Server，GuoLaiCode 就能用。

MCP 的核心思想是把工具的定义和实现分离。GuoLaiCode 作为 MCP Client，在启动时跟每个 Server 握手，获取可用工具列表。这些外部工具和内置工具在 Agent 眼里没有任何区别，都是工具列表里的一项，都通过 function calling 协议调用。

![]()

如果说内置工具是 GuoLaiCode 天生的六个手指，MCP 就是给它接上了无限多的延伸臂。每条延伸臂都是即插即用的，拔掉一条不影响其他的。

***

## 学会省钱和记事

能力有了，安全有了，但有两个实际问题开始浮出水面。

第一个问题：贵。

每次调用 LLM 都要花钱，按 token 计费。Agent Loop 每轮循环都要把完整的对话历史发给 Claude，随着对话越来越长，每次请求的 token 数线性增长。做一个复杂任务可能需要几十轮循环，到后面每轮请求都带着几万 token 的历史消息，大部分是之前的工具结果，对当前决策可能已经没什么用了。

![]()

第 8 章用两层策略解决了这个问题。

第一层是大结果落盘。工具返回的内容如果超过阈值，就写到磁盘上，messages 里只保留一个文件路径的引用。后续如果模型需要这个内容，再用 ReadFile 去读。这招简单粗暴但效果立竿见影，因为工具结果往往是最占 token 的部分。

第二层是自动压缩。当对话历史的 token 数超过窗口的一定比例，系统自动把整段对话发给 LLM 做一次摘要，用精炼的总结替换掉冗长的原文。压缩之后在 messages 里插入一个边界标记，告诉 Agent 哪些文件的内容可能已经过时，需要重新读取。

两层策略自动运行，用户无需干预。此外还有一个手动补充： `/compact` 命令让用户随时可以主动触发压缩，配合 token 用量报告，精确控制成本。

![]()

第二个问题：失忆。

每次启动 GuoLaiCode，它都是一张白纸。上一次对话里你跟它说过的偏好、项目的技术栈选型、之前踩过的坑，全部忘得干干净净。你不得不每次都重新交代一遍。

第 9 章给了 GuoLaiCode 三种记忆。

GUOLAICODE.md 指令文件是最持久的记忆。它在每次会话启动时被加载到 system prompt 里，相当于给 Agent 植入了一套固定的行为准则。项目级的约定、代码规范、特殊注意事项，写在这里就一劳永逸。

会话持久化是中期记忆。每次对话的完整消息历史以 JSONL 格式保存到磁盘，下次可以恢复。就像你离开工位时把笔记本合上，回来打开还是同一个页面。

自动记忆提取是最灵巧的一种。Agent 在工作过程中发现了值得记住的信息，比如项目的特殊约定、用户的偏好、反复出现的模式，就自动写入 memories.md。下次启动时，这些记忆被注入到上下文里，Agent 就能基于过去的经验做更好的决策。

![]()

***

## 从能用到好用

到第 9 章为止，GuoLaiCode 的核心能力已经完整了。它能自主工作，有安全保障，能控制成本，能记住经验。功能上已经够用了。

但够用和好用之间，还有一段不短的距离。

你有没有过这样的体验？一个软件功能很强大，但每次用都要翻文档才知道怎么操作。另一个软件可能功能没那么多，但用起来特别顺手，想做什么直觉就能找到入口。前者是能用，后者是好用。

第 10 章到第 12 章，就是把 GuoLaiCode 从能用推向好用的三章。

![]()

### Slash 命令：快捷通道

第 10 章实现了 Slash 命令框架。在此之前，你跟 GuoLaiCode 的所有交互都要通过自然语言，由 Agent Loop 处理。想压缩上下文？你得说「请压缩一下对话历史」，然后 Agent 调用对应的逻辑。想看当前状态？你得问「现在 token 用了多少」，等 Agent 去查。

Slash 命令把这些高频操作变成了直达快车。输入 `/compact` ，直接触发压缩，不经过 Agent Loop，不消耗 token。输入 `/status` ，直接显示当前状态。输入 `/plan` ，切换到规划模式。输入 `/help` ，看所有可用命令。

这是一条拦截快车道：用户输入以 `/` 开头时，命令框架直接接管，跳过 LLM 调用，执行对应的处理函数。快、省、确定性强。

![]()

### Skill 系统：可复用的技能包

第 11 章引入了 Skill 系统。Slash 命令是硬编码在程序里的，你想加一个新命令就得改代码、重新编译。Skill 换了一种思路：用 YAML + Markdown 描述一个技能，运行时动态加载。

Skill 的载体就是 `.md` 文件，frontmatter 里定义名称、触发条件、允许的工具列表，正文就是注入给 LLM 的提示词。用户可以自己写 Skill，放到指定目录，GuoLaiCode 启动时自动发现并加载。

这跟 MCP 的思路异曲同工。MCP 让工具可插拔，Skill 让行为可插拔。GuoLaiCode 的能力不再被编译时的代码限制，用户可以在运行时无限扩展。

![]()

### Hook 系统：生命周期自动化

第 12 章是好用三部曲的最后一环。Slash 命令是用户主动触发的，Skill 是用户主动调用的。Hook 解决的是另一个问题：有些事情你希望它自动发生，不需要你每次都记着去做。

比如每次 Agent 要执行 Bash 命令之前，你想自动检查命令是否安全。比如每次会话结束时，你想自动生成一份工作摘要。比如每次 Agent 修改了 `package.json` ，你想自动跑一次 `npm install` 。

Hook 系统定义了 15 个生命周期事件，覆盖了 Agent 运行过程中的关键节点。你可以在配置文件里声明：在某个事件发生时，如果满足某个条件，就执行某个动作。动作可以是跑一个 shell 命令，可以是发一个 HTTP 请求，可以是注入一段提示词，甚至可以是启动另一个 Agent。

![]()

三个系统各管一件事，加在一起让 GuoLaiCode 变得可以深度定制，真正贴合你自己的工作节奏。

***

## 从一个人到一支团队

前面十二章，GuoLaiCode 始终是一个 Agent 在工作。不管任务多复杂，都是它一个人扛。

但有些任务，一个人真的搞不定。或者说，搞得定，但太慢了。

想象你要重构一个大型项目的四个模块。如果只有一个 Agent，它得串行处理：改完模块 A，再改模块 B，再改 C，再改 D。但模块 A 和 B 的改动互不依赖，完全可以同时进行。

第 13 章到第 15 章，用三步把 GuoLaiCode 从单兵作战升级到了团队协作。

![]()

### SubAgent：学会委派

第 13 章让主 Agent 可以把子任务派给 SubAgent。每个 SubAgent 有独立的上下文窗口，干完活把结果交回来。这解决了上下文污染的问题：复杂任务拆成多个子任务，每个子任务在干净的环境里执行，不会互相干扰。

但 SubAgent 有两个局限。第一，子 Agent 之间不能直接通信，所有信息都得经过主 Agent 中转。第二，多个子 Agent 如果同时修改代码，文件冲突怎么办？

### Worktree：文件隔离

第 14 章用 Git Worktree 解决了文件冲突。每个子 Agent 在自己的 Worktree 里工作，拥有独立的文件系统副本。你改你的，我改我的，物理上就不可能冲突。等各自改完了，再合并回主分支。

Worktree 的巧妙之处在于它利用了 Git 已有的能力。不需要发明新的隔离机制，不需要复制整个项目目录，Git 帮你管理一切。

### Agent Teams：真正的协作

第 15 章是多 Agent 能力的终极形态。SubAgent 像一次性的外包，干完活就走。Agent Team 则持续运作，队员之间通过共享任务列表和 SendMessage 协调工作。

Team Lead 负责拆任务和分配，队员自主认领和推进。一个队员发现了影响其他人的变更，可以直接发消息通知。队员完成任务后可以暂停，需要的时候再被唤醒，带着完整的上下文继续工作。

复杂场景下，Lead 还可以开启 Coordinator Mode，主动放弃代码操作能力，专注调度。就像一个项目经理，不自己写代码，全部精力放在协调团队上。

![]()

三章下来，隔离上下文、隔离文件系统、建立协作通道，一层叠一层，GuoLaiCode 从一个人干活变成了能组团队。

***

## 站远了看一眼：完整架构

还记得第 1 章那张五层架构图吗？当时你可能觉得它只是一张抽象的蓝图。十五章各讲各的，现在把它们拼回去，你会发现每个格子都有了具体的血肉。

GuoLaiCode 的完整架构可以分成五层。从上到下：交互层（CLI/配置/UI/命令/技能）、引擎层（LLM 客户端/Agent 循环/编排）、工具层（内置工具/MCP/Hook）、记忆层（上下文压缩/会话管理/指令文件）、安全层（权限控制/路径沙箱/隔离）。安全层贯穿所有其他层，它包裹着整个系统，拦截每一次工具调用。

![]()

每一层只管自己的事，通过清晰的接口跟其他层通信。交互层不知道 Agent Loop 跑了几轮，引擎层不知道 UI 长什么样。工具层不关心是内置工具还是 MCP 工具，对 Agent 来说都一样。安全层拦截所有工具调用，但不干预具体的业务逻辑。

这种分层的好处在整个开发过程中你应该已经体会到了。每一章实现一层或一个子系统，都不需要动其他层的代码。加 MCP 支持的时候，Agent Loop 一行没改。 加 Hook 系统的时候，引擎层一行没改 。加 Agent Teams 的时候，权限系统一行没改。

![]()

### 贯穿始终的设计原则

回顾这十五章，有几个设计原则反复出现。它们贯穿整个系统，是所有章节共享的底色。

**事件驱动解耦。&#x20;**&#x4ECE;第 4 章的 AgentEvent 开始，GuoLaiCode 就建立了「生产者发事件、消费者处理事件」的模式。Agent 发出事件，UI 自己决定怎么响应，两边互不知道对方的实现细节。

这个模式在后续章节反复出现：Hook 系统监听生命周期事件，队员通过 mailbox 投递消息，Coordinator 通过 task-notification 获取结果。

**工具即能力边界。&#x20;**&#x41;gent 能做什么，完全由它的工具列表决定。权限系统禁用某些工具就能限制行为，Plan Mode 移除写工具就切换了模式。Coordinator Mode 和 Skill 的 tool allowlist 也是同一个思路。

同一个 Agent Loop，换一套工具集，就是一个完全不同的 Agent。

**LLM 做决策，系统做执行。&#x20;**&#x41;gent Loop 不做任何业务判断。该用什么工具、先做还是后做、遇到错误怎么调整，全由 LLM 决定。系统只负责忠实地执行工具调用、收集结果、传回给 LLM。

**渐进增强。&#x20;**&#x6BCF;一章都在前一章的基础上叠加能力，从不推翻重来。Agent Loop 是一个 while 循环，后来加了停止条件、事件流、Plan Mode，但核心循环从没变过。工具系统最初只有六个内置工具，后来加了 MCP，但工具的调用方式从没变过。

![]()

## 你还能做什么

GuoLaiCode 是一个完整的 Coding Agent，但它也是一个起点。你在这十五章中学到的架构思维，可以迁移到很多方向。

最直观的尝试是换一个 LLM。GuoLaiCode 的所有逻辑都不依赖 Claude 的特有能力，把 LLM Client 换成 OpenAI、DeepSeek、本地模型，Agent Loop 和工具系统完全不用动。你甚至可以让不同的 SubAgent 使用不同的模型，按任务特点分配。

引擎能换，场景当然也能换。Coding Agent 只是 Agent 的一种应用。把六个代码工具换成读 CSV、查数据库、画图表，你就有了一个 Data Agent。换成查日志、检查服务状态、执行部署，就成了 DevOps Agent。Agent Loop、权限系统、上下文管理、记忆系统，这些通用模块全都可以复用。

场景换了，工具自然也会越加越多。MCP 已经打开了大门，社区里有大量现成的 MCP Server，接上就能用。你也可以自己写 MCP Server，把公司内部的 API 包装成工具给 Agent 调用。

再往深处走，还可以在调度策略上做文章。目前的 Agent Teams 让 LLM 自己推理如何分配任务，你可以在此基础上加入更结构化的方案。比如根据任务的文件依赖关系自动构建 DAG，按拓扑排序安排执行顺序。比如根据每个队员过去的表现，动态调整任务分配。

![]()

以上是你现在就能动手的方向。但 Agent 领域本身也在快速变化，有几个趋势值得留意。

第一个趋势是上下文窗口越来越大。Claude 已经支持 200K token 的窗口，未来还会更长。你可能会问：窗口够大了，第 8 章的压缩策略还有意义吗？答案是仍然有。窗口变大解决的是「能不能装下」的问题，压缩解决的是「该不该装下」的问题。把三万行无关的工具输出塞进上下文，模型不会报错，但它的注意力会被稀释，决策质量会下降。大窗口让你有更多余量，但精细的上下文管理永远不会过时。

第二个趋势是 Computer Use。模型不再局限于调 API 和读写文件，而是可以直接操作图形界面：点击按钮、填写表单、截屏理解页面内容。这会改变工具设计的思路吗？会，但不是替代，而是补充。GUI 操作适合那些没有 API 的场景，比如操作一个只有 Web 界面的内部系统。但凡有 API 可调，结构化的工具调用在速度、可靠性和成本上都远胜于截屏加点击。GuoLaiCode 的工具层架构刚好能容纳这种扩展：Computer Use 就是另一个工具，插进去就行。

第三个趋势是多 Agent 协作正在从实验走向工程化。现在的难点不在于能不能让多个 Agent 同时工作，而在于怎么让它们高效协作而不互相扯后腿。上下文隔离、文件隔离、任务分配、冲突合并，这些你在第 13 到 15 章已经实现了基础版本。但现实中的挑战更复杂：Agent 之间怎么共享中间发现？怎么处理一个 Agent 的改动推翻了另一个 Agent 的前提假设？怎么在成本和质量之间找到平衡？这些问题没有标准答案，但你已经有了思考它们的框架。

这些方向没有标准答案，也不需要全部都做。选一个你感兴趣的，动手试试。你已经有了全部的基础知识，剩下的就是在实践中打磨。

***

## 结语

回到最开头的问题：什么是 Coding Agent？

第 1 章给了一个定义：LLM 在循环中根据环境反馈自主使用工具的系统。

十五章之后，这句话对你来说应该不再是一个抽象概念了。LLM 就是那个每轮循环都在推理和决策的大脑。工具就是 ReadFile、WriteFile、Bash，加上 MCP 接入的一切外部能力。循环就是那个 while 循环，简单到让人不敢相信它能驱动这么复杂的系统。反馈就是工具执行的结果，驱动着下一轮决策。

你亲手把这四样东西组装到了一起，然后在它上面加了安全层、加了记忆、加了命令和技能、加了多 Agent 协作。一层一层地，从一个想法变成了一个完整的系统。

![]()

Agent 的世界才刚刚开始。模型在变强，工具在变多，协作模式在进化。你在这门课里学到的架构思维和设计原则，会帮助你理解和驾驭未来出现的每一种新范式。

因为万变不离其宗：LLM + 工具 + 循环 + 反馈。

去造点什么吧。
