## 1. Agent Loop 与 ReAct 范式

### 原理篇：理论学习：ReAct 范式与 Agent Loop

**核心概念**：
- ReAct 范式：Think → Act → Observe 三步循环
- Agent Loop：while 循环驱动 LLM 反复思考和行动
- 停止条件：模型主动结束、迭代上限、用户取消、异常检测
- AgentEvent 流：事件驱动架构，UI 实时看到 Agent 在干什么

### 代码篇：Python源码解析：Agent 主循环与事件流

**关键实现位置**：`guolaicode/agent.py`

#### 1.1 AgentEvent 类型定义

**实现技巧**：用 Union 类型模拟代数类型

```python
@dataclass
class StreamText:
    text: str

@dataclass
class ToolResultEvent:
    tool_id: str
    tool_name: str
    output: str
    is_error: bool
    elapsed: float

AgentEvent = (
    StreamText | ThinkingText | RetryEvent
    | ToolUseEvent | ToolResultEvent | TurnComplete
    | LoopComplete | UsageEvent | ErrorEvent
    | PermissionRequest | CompactNotification | HookEvent
)
```

**注意事项**：
- 12 种事件覆盖 Agent Loop 的所有通知场景
- `CompactNotification` 携带 `boundary` 字段用于持久化压缩边界记录
- `PermissionRequest` 用 `asyncio.Future` 实现反向通信，UI 和 Agent 完全解耦

#### 1.2 主循环骨架

**实现技巧**：async generator + 惰性推送

```python
async def run(
    self, conversation: ConversationManager
) -> AsyncIterator[AgentEvent]:
    while True:
        iteration += 1
        if iteration > self.max_iterations:
            yield ErrorEvent(...)
            break
        # 1. turn_start hook
        # 2. 消费通知队列
        # 3. 自动上下文压缩
        # 4. pre_send hook
        # 5. 构建系统提示词
        # 6. Plan Mode 注入
        # 7. 获取工具 schema，调 LLM
        # 8. 消费流式响应
        # 9. max_tokens 恢复
        # 10. 没有工具调用 → 结束
        # 11. 分批执行工具
        # 12. 连续未知工具检查
        # 13. turn_end hook
```

**注意事项**：
- `run()` 本身是 async generator，调用方直接 `async for event in agent.run(conv)` 消费
- 在每个关键节点穿插 Hook 调用，覆盖 Agent Loop 的完整生命周期
- 流消费结束后统一执行工具，而不是在流式阶段就提交，让控制流更清晰

#### 1.3 StreamCollector：流消费器

**实现技巧**：流的读取逻辑和主循环彻底解耦

```python
class StreamCollector:
    def __init__(self) -> None:
        self.response = LLMResponse()

    async def consume(self, stream: AsyncIterator[StreamEvent]) -> AsyncIterator[AgentEvent]:
        async for event in stream:
            if isinstance(event, TextDelta):
                self.response.text += event.text
                yield StreamText(text=event.text)
            elif isinstance(event, ToolCallComplete):
                self.response.tool_calls.append(event)
                yield ToolUseEvent(...)
            elif isinstance(event, StreamEnd):
                self.response.stop_reason = event.stop_reason
```

**注意事项**：
- `consume()` 是 async generator：接收 LLM 流式事件，内部做汇总，同时把转换后的 AgentEvent 往外 yield
- 主循环只需要 `async for event in collector.consume(llm_stream): yield event` 一行
- 整个流处理链是懒的：LLM 产一个 token → collector 收集 + 转换 → 主循环 yield → UI 消费

#### 1.4 终止判断与记忆提取

**实现技巧**：fire-and-forget 后台任务

```python
if not response.tool_calls:
    conversation.add_assistant_message(
        response.text, thinking_blocks=conv_thinking)
    self._loop_count += 1
    if (self._loop_count % MEMORY_EXTRACTION_INTERVAL == 0
        and self.memory_manager):

            self._extract_memories(conversation))
```

**注意事项**：

```python
def is_safe_command(command: str) -> bool:
    trimmed = command.strip()
    if not trimmed:
        return False
    for ch in ("|", ";", "&&", ">", "$(", "`"):
```

## 3. 工具注册与执行框架

### 原理篇：理论学习：Function Calling 与工具系统

**核心概念**：
- Function Calling 四步：告诉模型有哪些工具 → 模型决定调用 → 执行工具返回结果 → 模型继续
- 工具描述是最值得打磨的部分
- 工具 Schema 自动生成

### 代码篇：Python源码解析：工具注册与执行框架

**关键实现位置**：`guolaicode/tools/`

#### 3.1 Tool 接口设计

**实现技巧**：ABC 抽象基类 + Pydantic 参数模型

```python
class Tool(ABC):
    name: str
    description: str
    params_model: type[BaseModel]
    category: ToolCategory = "read"
    is_concurrency_safe: bool = False
    is_system_tool: bool = False
    should_defer: bool = False

    @abstractmethod
    async def execute(self, params: BaseModel) -> ToolResult: ...

    def get_schema(self) -> dict[str, Any]:
        schema = self.params_model.model_json_schema()
        schema.pop("title", None)
        return {
            "name": self.name,
            "description": self.description,
            "input_schema": schema,
        }
```

**注意事项**：
- 元字段不是摆设：`category` 给权限系统判断，`is_concurrency_safe` 给执行引擎决定并发，`should_defer` 给注册中心判断默认隐藏
- `params_model` 挂一个 Pydantic 模型，Schema 就能自动生成
- `model_json_schema()` 把模型字段直接转成标准 JSON Schema，`pop("title")` 去掉自动塞的标题字段

#### 3.2 ToolResult 类型

**实现技巧**：用 dataclass 而不是 Pydantic BaseModel

```python
@dataclass
class ToolResult:
    output: str
    is_error: bool = False
```

**注意事项**：
- ToolResult 是内部数据结构，不需要序列化和校验的开销
- `is_error` 不是程序异常，而是告诉模型「这次没成功」，模型会重新判断再试

#### 3.3 ToolRegistry 注册中心

**实现技巧**：字典保持插入顺序，运行时屏蔽和延迟激活

```python
class ToolRegistry:
    def __init__(self) -> None:
        self._tools: dict[str, Tool] = {}
        self._disabled: set[str] = set()
        self._discovered: set[str] = set()

    def register(self, tool: Tool) -> None:
        self._tools[tool.name] = tool

    def get_all_schemas(self, protocol: str = "anthropic") -> list[dict[str, Any]]:
        schemas: list[dict[str, Any]] = []
        for name, tool in self._tools.items():
            if name in self._disabled:
                continue
            if getattr(tool, "should_defer", False) and name not in self._discovered:
                continue
            base = tool.get_schema()
            # 按协议适配输出格式
            if protocol in ("openai", "openai-compat"):
                schemas.append({"type": "function", "name": base["name"],
                    "description": base["description"], "parameters": base["input_schema"]})
            else:
                schemas.append(base)
        return schemas
```

**注意事项**：
- 字典保持插入顺序，所以注册顺序就是呈现顺序
- `_disabled` 集合负责运行时屏蔽，`_discovered` 跟踪哪些延迟工具已经被 ToolSearch 激活
- 查找靠 `get(name)` 返回 `Tool | None`，Schema 生成做两层过滤加一层协议适配

#### 3.4 默认注册工厂

**实现技巧**：延迟导入避免循环依赖，依赖注入只传给需要的工具

```python
def create_default_registry(file_cache=None, file_history=None):

    registry = ToolRegistry()
    registry.register(ReadFile(file_cache=file_cache, file_state_cache=file_state_cache))

    registry.register(EditFile(file_cache=file_cache, file_history=file_history,
                                file_state_cache=file_state_cache))
```

**注意事项**：

- `file_cache`、`file_history` 和 `file_state_cache` 通过依赖注入只传给需要的工具

```python
def persist_tool_result(tool_use_id: str, content: str, session_dir: Path) -> Path:
    file_path = session_dir / f"{tool_use_id}.txt"
    try:
        fd = os.open(str(file_path), os.O_WRONLY | os.O_CREAT | os.O_EXCL)
        with os.fdopen(fd, "w", encoding="utf-8") as f:
            f.write(content)
    except FileExistsError:
        pass
    return file_path
```

## 6. System Prompt 组装管线

### 原理篇：理论学习：System Prompt 如何设计？

**核心概念**：
- System Prompt 的本质：约束模型的默认倾向，限制行为范围
- 七个模块：角色设定、行为准则、工具使用指南、代码质量规范、禁止事项、任务策略、环境信息
- 原理：角色越聚焦，行为越可预测

### 代码篇：Python源码解析：系统提示词组装管线

**关键实现位置**：`guolaicode/prompts/system_prompt.py`

#### 6.1 PromptSection：提示词片段

**实现技巧**：dataclass + priority 排序

```python
@dataclass
class PromptSection:
    name: str
    priority: int
    content: str
```

**注意事项**：
- `priority` 决定各段落在最终 System Prompt 中出现的顺序
- 数字越小越靠前，Identity（priority=0）永远排在第一位，Memory（priority=95）排在最后
- 刻意留了间隔，方便后续在两个段落之间插入新内容

#### 6.2 PromptBuilder：组装器

**实现技巧**：链式调用 + 延迟排序

```python
class PromptBuilder:
    def __init__(self) -> None:
        self._sections: list[PromptSection] = []

    def add(self, section: PromptSection) -> PromptBuilder:
        self._sections.append(section)
        return self

    def build(self) -> str:
        self._sections.sort(key=lambda s: s.priority)
        parts = [s.content.strip() for s in self._sections
                 if s.content.strip()]
        return "\n\n".join(parts)
```

**注意事项**：
- `add()` 返回 `self`，支持链式调用
- `build()` 调用时才排序，可以以任意顺序 `add`
- 排序用 `list.sort()`，`key=lambda s: s.priority` 提取排序键
- 用列表推导式过滤掉空内容，最后用 `"\n\n".join()` 拼接

#### 6.3 build_system_prompt：入口函数

**实现技巧**：关键字参数 + 默认值代替 Builder 模式


- 七个全局常量 + 一个动态生成的环境段落，priority 从 0 到 70，间隔为 10
- `environment_section(work_dir)` 是函数调用而不是全局常量，因为工作目录、平台信息每次都不一样

#### 6.6 条件段落：80 到 95

**实现技巧**：用户自定义内容排在系统指令之后

```python
if custom_instructions:
    b.add(PromptSection(
        name="CustomInstructions",
        priority=80,
        content=(
            "# Project Instructions\n\n"
            + custom_instructions
        ),
    ))

if skill_section:
    b.add(PromptSection(
        name="Skills", priority=90,
        content=skill_section,
    ))

if memory_section:
    b.add(PromptSection(
        name="Memory", priority=95,
        content=memory_section,
    ))
```

**注意事项**：
- 三个条件段落只有在内容非空时才加入
- 优先级 80、90、95 放在固定段落之后，是用户自定义或运行时注入的内容
- 排序逻辑：项目指令 → 技能描述 → 记忆
- 记忆排最后是因为它最接近对话上下文，放在 System Prompt 尾部让 LLM 更容易「记住」

#### 6.7 Hook 注入：最后的拼接

**实现技巧**：字符串拼接不参与优先级排序

```python
result = b.build()

if hook_prompts:
    result += ("\n\n# Hook Injected Context\n"
               + "\n".join(hook_prompts))

return result
```

**注意事项**：
- Hook 注入的内容不走 PromptBuilder，直接字符串拼接在 `build()` 结果之后
- Hook 是外部系统注入的上下文，不参与优先级排序，永远排在最后
- 用字符串拼接比再包一层 PromptSection 更直接

## 总结：代码实现的核心技巧

通过以上对照分析，我们可以总结出 GuoLaiCode 代码实现的几个核心技巧：

### 1. 类型设计技巧
- **Union 类型模拟代数类型**：AgentEvent、DecisionEffect
- **Literal 限制取值范围**：ToolCategory、DecisionEffect
- **dataclass vs Pydantic**：内部数据结构用 dataclass，外部接口用 Pydantic
- **Enum 继承 str**：PermissionMode，枚举值可以直接当字符串用

### 2. 并发与异步技巧
- **async generator**：Agent Loop、StreamCollector
- **asyncio.create_task**：fire-and-forget 后台任务（记忆提取）
- **asyncio.Future**：反向通信（PermissionRequest）
- **文件锁并发控制**：Mailbox

### 3. 数据管理技巧
- **字典保持插入顺序**：ToolRegistry
- **多字典管理不同维度**：TeamManager
- **决策冻结**：ContentReplacementState，保持对话前缀稳定
- **浅拷贝**：clone_replacement_state，用于 fork 子 Agent

### 4. 工程实践技巧
- **依赖注入**：PermissionChecker、create_default_registry
- **延迟导入**：避免循环依赖（coordinator_mode）
- **原子写入**：os.O_EXCL 独占创建（persist_tool_result）
- **幂等设计**：文件名用 tool_use_id，天然幂等
- **关键字参数 + 默认值**：代替 Builder 模式（build_system_prompt）

### 5. 性能优化技巧
- **惰性推送**：async generator，消费者拉一个，生产者才产一个
- **流处理链**：LLM 产一个 token → collector 收集 + 转换 → 主循环 yield → UI 消费
- **Prompt Cache 稳定**：决策冻结，前缀逐字节不变
- **延迟排序**：PromptBuilder，build() 调用时才排序

### 6. 安全防护技巧
- **多层防御**：五层权限防御链
- **frozenset**：不可变集合，比普通 set 更安全
- **正则模式 + 原因描述**：危险命令黑名单
- **元字符检测用元组遍历**：一行代码搞定六种字符的检查
